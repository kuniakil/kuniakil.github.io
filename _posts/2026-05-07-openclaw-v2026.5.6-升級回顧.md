## 為什麼要整理 Workflow？

每次升級 OpenClaw 版本時，官方 repository 會攜帶大量的 GitHub Actions Workflows。這些 workflows 在 fork 的版本上執行時往往沒有實際用途，卻持續消耗 GitHub Actions 的 minutes quota。

這次升級的重點不在於新功能，而是**清理與最佳化**。

---

## 這次做了什麼

### 1. 建立備份分支

在開始任何改動之前，先建立了備份分支：

```bash
git branch backup/my-config-v2026.5.5
```

### 2. 從官方 v2026.5.6 創建新分支

```bash
git fetch upstream --tags
git checkout -b my-config-v2026.5.6 v2026.5.6
```

### 3. 套用自定義 commit

依序 Cherry-pick 過去的自定義修改：

- `chore: upgrade to v2026.4.30 baseline and restore essential configs`
- `docs: finalize v2026.5.2 upgrade and record official baseline strategy`
- `chore: remove redundant official workflows`
- `docs: update SOP to include mandatory workflow cleanup`

### 4. 清理 Workflows

刪除所有官方 workflows，只保留 `docker-release.yml`。這是發布 Docker image 到 GitHub Container Registry 的核心 workflow。

---

## 成果

- **分支隔離**：每個版本都有獨立的 branch，發生問題時可以快速回滾
- **節省 minutes**：移除不需要的 workflows，大幅降低 Actions 用量
- **乾淨的 Git History**：所有自定義修改都是明確的 commit，易於追蹤

---

## 如何回滾

如果 `my-config-v2026.5.6` 需要回滾：

```bash
# 方式一：切換回舊分支
git checkout my-config-v2026.5.5

# 方式二：強制推送舊分支到新分支
git checkout my-config-v2026.5.5
git push --force origin my-config-v2026.5.6
```
